# Solar

Solar is a custom programming language. The compiler is written in Rust (edition 2024).

## Dev environment setup

Required tools:
- **Rust** (via rustup)
- **LLVM and clang** — version must match rustc's LLVM version (`rustc --version --verbose | grep LLVM`). The distro repos may lag; the matching version usually comes from the official LLVM apt repo (`apt.llvm.org`, e.g. `llvm-toolchain-trixie-22`).
- **LLVM dev headers + `clang++` and `llvm-config`** — `build.rs` compiles the GC write-barrier LLVM pass plugin (`llvm-pass/SolarWriteBarriers.cpp`) against these. Install e.g. `llvm-22-dev`. Without them `cargo build` still works (interpreter only) but native codegen fails with a clear message.
- **lld** — same version as LLVM/clang
- **Node.js** — needed by tree-sitter to generate the parser
- **tree-sitter CLI** — `cargo install tree-sitter-cli`

The unversioned commands (`clang`, `clang++`, `llvm-as`, `llvm-link`, `llc`, `opt`, `ld.lld`, `llvm-config`) must be on `$PATH`.

## Build

```
cargo build
```

## Run

```bash
# uses IR interpreter, does not support multithreaded features
cargo run -- examples/example.solar

# script adds path and file extension, this runs `examples/my_file.solar`
./run_example.sh my_file

# compile and run any solar file at any path
cargo build --release -p solar-system # build the runtime libaries for linking in release mode, since the next command will use the optimized version
cargo run --bin compile -- my_path/my_file.solar target/my_executable # omits `--release` to make rustc faster, but the solar compiler will run in release mode
target/my_executable # execute the new program

# compile and run in debug mode (enables asan)
cargo run --bin run_codegen -- my_path/my_file.solar
```

The intended use is to compile in release mode using `--bin compile`, any other uses are for debugging.

### Diagnostic flags (environment variables)

These flags only apply to compiled binaries, and not the interpreter.

- `SOLAR_PRINT_GC_STATS=1` - prints stats for each GC cycle
- `SOLAR_PRINT_ALLOCS=1` - prints EVERY allocation; this may flood the output, so use sparingly!
- `SOLAR_DISABLE_GC=1` - disables GC and switches to a bump allocator

## Changing syntax rules

When modifying the grammar, follow all of these steps:

1. Edit `tree-sitter-solar/grammar.js`
2. Update `src/parser.rs` — CST-to-AST conversion (field names, node kinds)
3. Update `examples/example.solar` to use the new syntax
4. Build and test: `cargo build && cargo run -- examples/example.solar`

The C parser is regenerated automatically by `tree-sitter-solar/build.rs` when `grammar.js` changes. Do not edit `tree-sitter-solar/src/` by hand — it is generated and gitignored.

## Pipeline

The pipeline is orchestrated through `src/pipeline.rs` with typed stages and method chaining:

```
pipeline::compile(path) → Typed → .to_ir() → Ir → .to_c(name) → CSource → .to_binary(name, mode) → Binary → .run(name) → String
```

Entry point is `pipeline::compile(file_path)` which returns a `Typed` struct. Each stage wraps its data and has methods to advance to the next stage. You can stop at any stage (e.g., stop at `Typed` for typecheck tests, at `Ir` for interpreter tests).

`CSource::to_binary` supports two modes via `CompileMode::Debug` (ASAN + clang, links `target/debug/libsolar_system`) and `CompileMode::Release` (LLVM LTO, cross-language optimization, allocator attribute stamping, links `target/release/libsolar_system.a`). Intermediate files go in `target/solar/{name}_{random_hex}/` and are kept for debugging. In release, only `solar_system`'s **own** bitcode (its `*.rcgu.o` codegen units plus the assembled `atomic128.o`) is `llvm-link`ed into the module that gets re-optimized with the user's code — that's what needs cross-language inlining and the GC alloc/barrier passes. The runtime's dependency crates (backtrace, gimli, addr2line, object, rustix, …) are ~11 MB of the ~15 MB of bitcode in the archive but are panic/backtrace-only, never hot, never inlined, and already fat-LTO-optimized when cargo built the archive; merging them into the per-compile `opt -O3` was the dominant cost (~4× slower). They now resolve from the `.a` at the final link instead. Because those archive members are bitcode (fat LTO) and are now pulled at link time, the final `clang` link uses `-fuse-ld=lld` (GNU ld can't read bitcode archive members; lld LTO-compiles them). The archive only contributes members for symbols not already defined in the merged module, so there's no duplicate-symbol conflict with the `solar_system` bitcode already linked in.

GC write barriers are inserted by a real LLVM pass plugin (`llvm-pass/SolarWriteBarriers.cpp`), built by `build.rs`. The plugin provides two passes run via `opt -load-pass-plugin=…`:

- **`solar-lower-gc-alloc`** (release only, *before* `opt -O3`): rewrites each generated `sol_alloc(size, align, mark_fn)` call into a recognized `calloc(1, size)` carrying `(align, mark_fn)` in `!solar.alloc` instruction metadata, and each `sol_memcpy` into `llvm.memcpy`. This is the key to allocation elision: LLVM's allocation-promotion/dead-alloc transforms key on *recognized* allocators (TargetLibraryInfo), so they SROA/elide a non-escaping `calloc` but **not** a custom `sol_alloc`, even one stamped with the full malloc attribute set. `calloc` (not `malloc`) preserves `sol_alloc`'s zeroing so the optimizer never sees uninitialized reads. The referenced `_mark_*` functions are anchored in `llvm.compiler.used` so they survive globaldce until raising.
- **`solar-write-barriers`** (debug + release, *after* `opt -O3`): first *raises* surviving `calloc … !solar.alloc` calls back to `sol_alloc` (reading the metadata), then inserts the barriers. It instruments `store ptr` (and `<N x ptr>` vector stores → bulk barrier) into non-stack/global destinations, plus aggregate copies — `llvm.memcpy`/`memmove` and any residual `sol_memcpy` call — via `sol_gc_memcpy_barrier`. `getOrInsertFunction` declares the barrier runtime functions when the module doesn't define them.

Inserting barriers as IR (not text — replacing an earlier `llvm-dis | edit | llvm-as` rewrite) gives robust stack-vs-heap provenance via `getUnderlyingObject`, native debug-location propagation (the inserted call inherits the store's `!dbg`, so LLVM never strips module debug info — which had silently dropped solar-system DWARF for profilers), and type safety. Both barrier-relevant runtime side effects are kept out of the optimizer's view until *after* elision: barriers are inserted post-`opt`, and `sol_memcpy` is a **plain copy** (no inline shading — that would make freshly-allocated objects escape and block their elision; the shading is re-added by the post-opt memcpy barrier). In release the final `clang -O3` link inlines the barrier fast path; in debug the pass runs on the `-emit-llvm` bitcode before the (LTO-deferred) ASAN instrumentation, so debug/ASAN test binaries also exercise the collector with barriers active.

The collector (`solar-system/src/gc.rs`) is **concurrent**: a dedicated GC thread spawned in `sol_start` owns every cycle. Mutators never collect — on the heap-growth heuristic `sol_alloc` calls `request_gc` (a futex wake) and keeps running. A cycle is: **STW pause 1** (signal all registered mutators, scan their stacks+registers, seed the gray queue with root pointer values, set `SOL_CONCURRENT_MARKING`, resume) → **concurrent parallel mark** (`parallel_mark` submits one job per thread-pool worker; workers drain the gray queue, follow heap edges, and overflow excess back to the queue for idle workers to steal, until quiescence — while mutators run, the Dijkstra insertion barrier and the inserted aggregate-copy barrier (`sol_gc_memcpy_barrier`) shade newly-stored pointers, and new allocations are *born black*) → **STW pause 2** (clear the flag, flush gray buffers + re-scan roots, drain to fixpoint, then the parallel sweep). The gray frontier is a **sharded** queue (`GRAY[N]` + per-thread buffers in `ThreadSlot`) kept separate from the alloc/mark bitmaps; consumers swap a whole shard in O(1). The barrier does **white-only shading** (skips already-marked targets via `heap::is_marked_addr`) so a fast allocator's born-black stores don't flood the queue. **Allocation back-pressure**: `ALLOCATED_SINCE_GC` (batched per-thread) is capped at `alloc_hard_cap()` (a floor, else scaled by *traced* live — live excluding this cycle's born-black float, which avoids a float→cap→float feedback loop); over the cap, `sol_alloc` stalls as a safepoint (`stall_for_gc`) until a cycle reclaims space. Remaining caveat: marker heap reads are racy-by-design (sound on x86 for aligned words).

**GC-managed file descriptors** (`solar-system/src/file.rs`): the built-in `FileDesc` type is an opaque handle with the byte representation of `&Int32` — a traced pointer into a separate **fd arena** (a 4 GiB `PROT_NONE` mmap that is *never read or written*; the fd number is recovered arithmetically as `addr - FD_BASE`). The `file_open(path, flags: Int, mode: Uint)` intrinsic (defined in `src/std/file.solar` and re-exported from `lib.solar`) calls `sol_file_open`, which `open(2)`s the file with the given flags (always OR-ing in `O_CLOEXEC`) and mode and returns `FD_BASE + fd`, setting the fd's bit in a side alloc bitmap. The `flags`/`mode` are an implementation detail of `@std`: `file::open(path, read=true, write=true, append=false, truncate=false, new=false, create=false, mode=0o666u)` takes boolean keyword args and builds the `open(2)` flag word from `O_*` bit constants declared in its own body (`new` → `O_EXCL`); `mode` (the creation permission bits) is a keyword arg defaulting to 0o666 (a second mark bitmap and an `FD_HWM` mirror the heap's bitmaps). The collector traces `FileDesc`s like any pointer: `plausible()` accepts fd-arena addresses, the generated mark function (`_mark_single_ptr`) enqueues the value, `drain` routes it to `file::fd_mark`, and the write barriers shade fd-arena stores (white-only via `file::is_marked`). At STW pause 2, `file::fd_sweep` runs alongside the heap sweep and `close()`s every fd whose slot is allocated-but-unmarked — i.e. no live `FileDesc` remains. Born-marked-during-mark mirrors born-black. The `file_close` intrinsic (`file::close(f)` in `@std`) calls `sol_file_close`, which does *not* `close()` the fd (that would free the number, letting a later `open` reuse it and a stale `FileDesc` silently alias the new file). Instead it `dup2`s a process-wide **dead fd** — the read end of a pipe whose write end is closed at startup, so reads return EOF — over the fd: the underlying file is released now, but the number stays occupied (and its alloc bit set, so the collector still traces and eventually reclaims it). The `file_stdin`/`file_stdout` intrinsics (`file::stdin()`/`file::stdout()` in `@std`) return a `FileDesc` for the process's standard streams (`sol_file_stdin`/`sol_file_stdout` return `FD_BASE + 0`/`FD_BASE + 1`); these are deliberately **not** registered in the alloc bitmap, so the collector traces them harmlessly but `fd_sweep` (which only closes allocated-but-unmarked fds) never auto-closes them. The `file_read(fd, dst: &[Uint8]) -> Uint`/`file_write_partial(fd, src: &[Uint8]) -> Uint` intrinsics (`sol_file_read`/`sol_file_write_partial`) call `read(2)`/`write(2)` directly (single, possibly-partial syscall, retried on `EINTR`, panic on other I/O error), and back the `FileDesc` methods in `file.solar`: `read(buf)` / `read(max_size)` (the latter allocates a GC buffer, reads once, and returns the bytes read as a `&[Uint8]` slice into it), `write_partial(src)`, and `write(src)` (loops `write_partial` to a full write, like Rust's `write_all`). There is no dedicated stdin/stdout intrinsic — `@std`'s `write_stdout`/`println` are `file::stdout().write(...)`, and stdin is read via `file::stdin().read(...)`. The interpreters have no fd arena/collector: a `FileDesc` is a plain integer index into a per-run virtual table (`src/interp_io.rs`, `FileTable` of `Box<dyn Read + Write>`) — index 0 is stdin, 1 is stdout (wrapped so the unsupported half panics), and `file_open` decodes the flag word into `OpenOptions` and pushes the boxed `File`, returning its index. `file_read`/`file_write_partial` route through this table; `interpret_to` takes both a `stdin` and `stdout`. `close` is a no-op in the interpreters (auto-close and dead-fd neutering are compiled-runtime only).

### Stage details

1. **Parse**: tree-sitter produces a CST, `parser.rs` converts it to an untyped `ast::SourceFile`.
   1b. **Resolve**: `resolve::resolve` recursively parses imported files, validates exports/visibility, and rewrites all ASTs into a single unified `ast::SourceFile` with module-mangled names (e.g., `foo::Point` → `__mod3_fooPoint`; the module segment is length-prefixed — see the mangling note in stage 2). Root file items keep their original names. The stdlib is parsed in the same resolver and every user file gets a synthetic `import * from "@std"` — stdlib pub items (`println`, etc.) are available directly, and pub module re-exports (e.g., `pub import vec from "vec.solar"` in lib.solar) become module aliases (e.g., `vec::Vec`). Wildcard imports propagate pub module re-exports from the source file. Import statements are stripped. Returns `(SourceFile, SourceMap)` for multi-file error reporting.
2. **Type check / lower**: `typed_ast::lower` walks the untyped AST, infers and checks types, and produces a `typed_ast::SourceFile` where every `Expr` carries a concrete `Type`. Panics on type errors. Closures are desugared into synthetic functions (`__closure_N`) with capture analysis; the `Closure` ExprKind records the synthetic function name and captured variables. Methods are desugared into regular functions with mangled names (`__method_` + the mangled name; the receiver type is the first argument and so the first mangled type arg); the receiver becomes the first argument. **Operator overloading**: a binary `a <op> b` whose left operand type has no primitive implementation of `<op>` (so `1 + 2` keeps the built-in path, but a struct/enum operand does not) is desugared in the `BinaryOp` arm into `a&.operator_<op>(b&)` and re-lowered through the ordinary method-call path — but only if a matching `operator_*` method is in scope (`operator_add`/`sub`/`mul`/`div`/`mod`/`eq`/`ne`/`lt`/`le`/`gt`/`ge`/`and`/`or`/`bitand`/`bitor`/`bitxor`/`shl`/`shr`; see `binop_method_name`/`binop_primitive_applies`). The receiver and argument are taken by reference, so an overload's parameters are typically `&T`. The lowered `typed_ast` therefore contains a `Call` (no new node kind — the schema is unchanged); overloaded ops never reach the `BinaryOp` arm of downstream layers. (The deref postfix `@` binds tighter than every binary operator — `deref_expr` precedence 70 in the grammar — so `self@.x * k@` parses as `(self@.x) * (k@)`.) **Bitwise operators** `&`/`|`/`^`/`<<`/`>>` (`ast::BinOp::BitAnd`/`BitOr`/`BitXor`/`Shl`/`Shr`) and the unary `!` (`ast::ExprKind::Not` → IR `NodeKind::Not`) are integer-only; their grammar precedence matches Rust (loosest→tightest: `|` < `^` < `&` < `<<`/`>>` < `+`/`-` < `*`/`/`/`%`, all below the postfix operators at 70, with unary `!` at 65 — looser than postfix, tighter than every binary op). The `&` and `^` tokens are shared with the postfix reference/unique operators, so the grammar declares GLR `conflicts` so tree-sitter picks the binary parse when a right operand follows and the postfix parse otherwise. For `&`/`|`/`^` **both operands must have the same type** (no implicit widening: `1u8 | 2u16` and `1u8 & 2i8` are type errors). Shifts are the exception: the **count (right operand) may be any integer type** — only its magnitude matters — so `word_u64 << i_uint` needs no cast (the same-type check is skipped for `Shl`/`Shr`; the count is validated as an integer separately). Shifts are **overflow-safe**: a count reaching or exceeding the type's bit width yields 0 (`<<` and unsigned `>>`) or the sign fill (signed `>>`); `>>` is arithmetic on signed types, logical on unsigned. **Wrapping arithmetic** is written with **doubled operators** — `++`/`--`/`**` (`ast::BinOp::WrapAdd`/`WrapSub`/`WrapMul`) — at the same precedence as the checked `+`/`-`/`*`. They are integer-only (no array concat), two's-complement, and **never panic on overflow** (the checked single-char operators still do); they wrap at the operand's width (`255u8 ++ 1u8 == 0u8`, `Uint::MAX ++ 1u == 0u`). `/` and `%` have no wrapping form — `//` is the line-comment syntax, and integer division overflows only in the `MIN / -1` edge case. Both interpreters compute wrapping on the raw 64-bit pattern (two's-complement add/sub/mul are sign-agnostic) and truncate to width; codegen wraps in the unsigned 64-bit domain and truncates via the cast to the result type. Unary `!` is logical-not on `Bool` and bitwise-complement on integers (result width-masked). The interpreters width-mask bitwise/shift results explicitly (`truncate_int` in `ast_interp`, `truncate_to_ty` in `ir_interp`) to match the truncation the compiled backend gets for free from the C cast to the result type, so e.g. `(!0u8) == 255u8` and `128u8 << 1u8 == 0u8` agree across all three backends. **Bit-counting intrinsics** `count_trailing_zeros`/`count_leading_zeros`/`count_ones` (`ast::Intrinsic`, `@intrinsics`) take any integer and return a `Uint` count over the operand's bit width (ctz/clz return the full width for a zero input). They're exposed as the generic stdlib methods `count_trailing_zeros#[T]`/`count_leading_zeros#[T]`/`count_ones#[T]` in `lib.solar` — methods *may* have a bare type-parameter receiver (`self: T`); the receiver type is inferred at the call site like any generic. Codegen lowers them to `__builtin_ctzll`/`clzll`/`popcountll` (→ `llvm.cttz`/`ctlz`/`ctpop` → a single `tzcnt`/`lzcnt`/`popcnt`), masking to width and guarding ctz/clz against the zero input; the interpreters use `u64::{trailing_zeros,leading_zeros,count_ones}` on the width-masked value. **`carrying_mul_add` intrinsic** (`ast::Intrinsic::CarryingMulAdd`, `@intrinsics`) is the one intrinsic with **`&` out-params**: its Solar-level signature is `carrying_mul_add(a: Uint64, b: Uint64, carry: Uint64, add: Uint64, out_lo: &Uint64, out_hi: &Uint64) -> Unit` — it computes the full 128-bit value `a*b + carry + add` (which never overflows 128 bits) and writes the low/high 64-bit halves through `out_lo`/`out_hi`. Codegen lowers it to the runtime function `sol_carrying_mul_add` (`solar-system/src/arith.rs`, next to `sol_checked_*_uint`), which does the arithmetic in `u128` and stores through the two `uint64_t*` pointers; the interpreters compute it inline (`u128` math, then store through the ref out-params). It's wrapped by the concrete (non-generic, `Uint64`-only) stdlib method `carrying_mul_add(self: Uint64, rhs, carry, add) -> (Uint64, Uint64)` in `lib.solar`, which allocates two locals, passes them as `lo&`/`hi&`, and returns `(low, high)`. **Little-endian decode intrinsics** `u64_from_le([Uint8; 8]) -> Uint64` and `u32_from_le([Uint8; 4]) -> Uint32` (`ast::Intrinsic::U64FromLe`/`U32FromLe`, `@intrinsics`, **intrinsics-only — no stdlib wrapper**) decode a fixed-size byte array as a little-endian integer. They take a `[Uint8; N]` *fixed array*, but the intended call passes a slice that coerces to it — `u64_from_le(s[i..i+8u])`: the slice is bounds-checked, and the `[Uint8]`→`[Uint8; N]` coercion (`ArraySizeCoerce`) asserts the runtime length is exactly `N`, so the read is always in-bounds and memory-safe (a wrong-length slice panics at the coercion, never reads OOB). To make this work, `lower_intrinsic_call`'s `Exact` param requirement now runs `try_coerce` before the type check (a no-op when types already match). Codegen materializes the array into a buffer and `__builtin_memcpy`s `N` bytes into the result (a byte copy IS the LE decode on the little-endian target; in release the buffer collapses to a single load); the IR interpreter reads `N` bytes via `Memory::load` (which uses `u64::from_le_bytes`); the AST interpreter folds the byte `Value`s with `<<`. **Name mangling** (`mangle_name`/`mangle_type` in `typed_ast.rs`) is **injective and reversible from the symbol alone** — distinct entities map to distinct C identifiers and the original (e.g. `lib::write_stdout(&[Uint8])`, `Box#[Int]`) can be recovered with no side table. The primitive is a self-delimiting identifier `enc_id(s) = "<byte-len>_<s>"`; type args are tagged (`R`=`&`, `Q`=`&?`, `U`=`^`, `S`=`[T]`, `A`/`F` for arrays/functions) with length-prefixed leaves, so a struct named `ref_Int` can't collide with `&Int`. `solar-system/src/panic.rs::demangle_solar` is the exact inverse (used to render panic backtraces) — **keep the two in sync**. **Optional keyword parameters** — `fn f(normal: Int, kw1: Int = 0, kw2 = 1u)` — must follow all normal parameters; their defaults must be literals (ints, bools, arrays, strings-as-arrays, and `&`/`^` of those), and an omitted type is inferred from the default (`prepare_keyword_params`, at function/method registration, bakes the inferred type and validates). They are **invisible to overload resolution** (overloads are matched on the required, non-default parameters only) and **keyword-only** at call sites (`f(1, kw1=2)` in `ast::ExprKind::Call`/`MethodCall`'s `kwargs`). `resolve_overloaded_call` expands each call into a fully positional argument list — provided kwargs by name, unspecified optionals from their default literal — so the lowered `typed_ast`/IR sees only ordinary positional calls (no kwarg concept downstream). Keyword args are rejected on non-registry calls (enum construction, nested, indirect). **Const declarations** (`const NAME [: T] = <literal>`, `ast::TopLevelItem::Const` / `StatementKind::Const`) bind a name to a literal value; they may be top-level (exported/imported like other items via `resolve`, module-mangled) or local to a block (scoped), and the type is optional (the explicit type, if given, coerces the literal). Each use site is **substituted with the (re-lowered) literal value** during `Identifier` lowering — innermost local const scope first (`const_scopes`, pushed/popped with `push_scope`/`pop_scope`), then top-level `consts` — so the lowered output contains no const declarations. The value must be a literal (`is_literal_default`). Generic structs/enums are monomorphized to an injective mangled identity: `Box#[Int]` becomes `3_BoxG1_3_Int`, `Option#[&Node]` becomes `6_OptionG1_R5_...` (see the mangling note above). **Nullable references** `&?T` are a distinct type (`Type::NullableRef`/`NullableRefUnsized`, mangled with the `Q` tag) with the same representation as `&T` (8-byte pointer, or 16-byte fat pointer for unsized `T`); their null value is written `null#[T]` (always with the type argument; `ExprKind::NullLiteral` → IR `NodeKind::Null` → a zero pointer). A normal `&T` implicitly coerces to `&?T` (a no-op retag in `try_coerce`); the reverse is **not** implicit — to get a `&T` from a `&?T`, write `nullable_ref@&` (the `@` null-checks and yields the pointee place, `&` re-borrows). Dereferencing `&?T` with `@` emits a null check (`sol_null_check` in compiled code, `assert!` in the interpreters; panic message `null pointer dereference`); `&T` deref is unchecked. `==`/`!=` compare a nullable ref against `null#[T]` (or another ref, at least one side nullable) by pointer identity. Destructuring patterns in `let` bindings and function parameters are desugared into temp variables + field accesses/indexing; downstream layers see only simple `let` statements. **`for` loops** (range and for-in) desugar into a `while` over a hidden counter/index; the counter increment is emitted **right after binding the loop variable and before the user body** so that `continue` (which re-checks the loop condition at the top) still advances it. **`loop { … }`** (`ast::ExprKind::Loop`) is an **expression**: it yields a value via `break <expr>` (only valueless `break`/`break <value>` distinguishes a `while`/`for`, which are statements, from a `loop`). Its type is the unified type of its break values (only valueless breaks → `Unit`; no breaks → `Never`). A bare `loop {…}` needs a trailing `;` as a statement (it's an expression) and is lowered through the statement path so `return`/outer `break` propagate. **`break`/`continue`** (`StatementKind::Break(Option<Expr>)`/`Continue`) are validated against a `loop_ctx` stack on the lowerer (an error outside any loop; tracks each loop's `is_value_loop` and unified break type), which is cleared when entering a closure or nested function so they can't escape into an enclosing function's loop; they lower to IR `NodeKind::Break(Option<NodeId>)`/`Continue` (and C `break;`/`continue;`, since `while` and `loop` both become `while(1){ … }`). A loop expression's value flows through a loop-result-destination stack (`loop_dst` in the IR interpreter and codegen) that `break <v>` writes into. The IR interpreter signals control flow via `ControlFlow::{Normal,Break,Continue,Return}` and the AST interpreter via a `Flow` enum. (Like other expression-position bodies, `return` inside a *value-position* `loop` is not propagated by the interpreters — only the compiled backend; bare-statement loops propagate it.) Compile-time reflection (`match.reflect Type { "struct" => ..., "enum" => ..., _ => ... }`) is resolved here: the inspected type is classified at compile time, the taken branch replaces the whole expression, and non-taken branches are erased without being type-checked. `for.reflect_fields x in o` (where `o: &T`, `T` a struct) unrolls its body into one scoped block per field with `x: (&[Uint8], &F)` — field name and value reference. `match.reflect_variant (variant, index, val) in o` (where `o: &T`, `T` an enum) desugars into a match over the enum with the body duplicated in every arm, binding the pattern against `(&[Uint8], Uint, Payload)` — variant name, the variant's 0-based discriminant index (a compile-time `Uint` literal per arm), and payload by value. A bare name binds the whole 3-tuple. For a unit variant under a `(variant, index, val)` tuple pattern, `val` binds to the **variant index** (a `Uint`) — not a payload, since there is none — so a generic body that references `val` (e.g. reflective hashing) compiles for enums that mix unit and data variants. **Paired reflection** — the distinct keywords `for.reflect_fields_pair` / `match.reflect_variant_pair` reflect two values of the *same* struct/enum in lockstep; their object must be a 2-tuple literal `(a, b)` (a distinct keyword, **not** an overloaded object form — so the plain `reflect_fields`/`reflect_variant` over a tuple still reflect that tuple value's own fields, identical to binding it to a variable first). `for.reflect_fields_pair (name, a, b) in (self, other)` binds `(&[Uint8], &F, &F)` — both objects' field references with matching static type, so `a@ == b@` typechecks even across heterogeneous sibling fields. `match.reflect_variant_pair (name, idx, sval, oval) in (self, other)` runs the body once per variant **only when both objects hold that variant** (mismatched variants are a no-op), binding both payloads with the same static type. These are the building blocks for the stdlib's blanket reflective `operator_eq#[T]` (in `lib.solar`, alongside the reflective `hash`): structural equality for any struct/enum (fields/payloads compared with `==`, byte-slice fields via a `&[Uint8]` `operator_eq` overload), so `a == b` and HashMap struct keys work with no hand-written impl. A concrete `operator_eq` is more specific and overrides the blanket. (Because the blanket is a `pub` stdlib method wildcard-imported everywhere, the resolver treats a name shared between an imported method and a local method as an overload set, not a conflict — see `resolve.rs` `local_method_defs`.) Downstream layers see only concrete mangled names.
3. **IR lower**: `ir::lower` converts the typed AST into a flat-tree IR. Variable names are erased to globally unique `VarId`s. Struct types get memory layouts (field offsets, sizes, alignment). Function bodies become flat `Vec<Node>` where children are referenced by `NodeId` index. Function values are 16 bytes (code pointer + env pointer). All Solar functions receive a hidden `__env` parameter; non-closures pass/ignore 0.
4. **Interpret (IR)**: `ir_interp::interpret` executes the IR using flat memory (`BTreeMap<address, u64>`). Structs are decomposed into per-field entries at layout offsets. Arrays allocate separate element storage (ptr+len). Refs are plain addresses. Unique pointers (`^T`) are also plain addresses but use type-aware deep copy: when a value containing `^T` is copied, the pointee is recursively cloned into fresh memory (unlike `&T` which copies only the pointer). All other copies are memcpy.
5. **Interpret (AST)**: `ast_interp::interpret` walks the typed AST directly and executes it using `Rc<RefCell<Value>>` slots. Kept as a reference interpreter; runtime tests assert both interpreters produce identical output.
6. **Compile (IR)**: `codegen::generate` lowers the IR into C code which is linked to `solar-system` to compile the program as a native executable. Runtime tests assert that the compiled program produces identical output as the interpreters.

## Project structure

- `src` — Main code
- `src/std/` — Standard library; `lib.solar` is the entry point, can import other files in this directory (including subdirectories — imports resolve as `base_dir.join(path)`, and `lib.solar`↔submodule cycles are fine, deduped by canonical path)
- `src/std/hashbrown/` — a memory-safe SwissTable `HashMap` (port of [hashbrown](https://github.com/rust-lang/hashbrown) + [foldhash](https://github.com/orlp/foldhash)); re-exported from `lib.solar` as `hashbrown::HashMap`. Modules: `tag`/`bitmask`/`group` (the SWAR control-byte machinery, using the bitwise/shift/wrapping ops, `count_trailing_zeros`, and `u64_from_le`), `foldhash` (the fixed-seed `Hasher`, **bit-exact** with upstream foldhash via `carrying_mul_add`), `hash` (`hash#[T]`: reflective for structs/enums, concrete overloads for primitives), `raw` (`RawTable`: probe/find/insert/resize/erase) and `map` (the public API: `new`/`insert`/`get`/`get_mut`/`contains_key`/`remove`/`len`/`is_empty`/`clear`/`for_each`). Deviations from hashbrown are memory-safety-driven and **no `unsafe`**: two GC allocations (`ctrl: &[Uint8]` always present + `slots: &?[Entry]` lazily allocated, null when empty — Solar can't make a generic array without a sample value, so slots are filled from the first inserted/any-existing entry, the `Vec`-style trick); bucket `i` is `slots[i]`; resize reallocs + reinserts. **Keys must be sized** (primitives, or structs/enums of sized fields); a struct/enum key gets content-based `==` and hashing for free from the stdlib's blanket reflective `operator_eq`/`hash` provided its fields are `pub` (no hand-written impl needed); unsized keys (bare `[T]` / strings) can't be stored by value. One Solar gotcha the port relies on: binding `let s = self@.slots@` **copies** the slice (arrays are value types), so to mutate the array you bind the *reference* and deref-index it — `let s = self@.slots; s@[i] = …` — or use the place chain `self@.slots@[i] = …` directly. (A separate interpreter bug — control flow not propagating out of a statement-position `if`/`match`/block expression — was fixed in `ir_interp`/`ast_interp`; both now run such a statement-position expression through `exec_body`/`exec_match_stmt` so `break`/`continue`/`return` escape correctly, matching codegen.)
- `tree-sitter-solar` — tree-sitter grammar: main file is in `grammar.js`
- `solar-system` — native library linked to compiled programs
- `examples` — example programs, main program is `example.solar
- `tests` — integration tests (runtime tests run both interpreters, typecheck error cases)
